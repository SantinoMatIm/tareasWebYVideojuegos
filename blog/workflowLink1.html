Flujo de trabajo de Git y Github

  1. Configuración inicial

    2. Se guarda la información del usuario y la identidad para que cada commit realizado quede registrado con nuestro username. Usa el comando git init para crear un repositorio en donde guardar la carpeta oculta .git

  2. Trabajo con repositorios remotos

    3. Se permite la conexión entre nuestro repositorio local y uno remoto en Github.
    Al clonar un repositorio, se descarga una copia completa del proyecto junto con el historial. Esto nos permite sincronizar cambios y trabajar de forma remota en nuestra propia versión.
    Los comandos usados son git clone.

  3. Flujo básico de trabajo

    4. Es el ciclo principal y fundamental de trabajo en Gi y Github. Con git status podemos ver el estado actual del repositorio, indicando que archivos han sido modificados o se encuentran en el staging area. Con git add seleccionamos los archivos que deseamos incluir al staging area. Con git commit se guardan los cambios en el historial del repositorio con un mensaje descriptivo

  4. Trabajo con branches

    5. Las branches sirven para trabajar en distintas tareas sin afectar el código principal. Con git branch se puede ver las ramas existentes en el repositorio y permite crear nuevas. git checkout y git switch también sirven para cambiar entre ramas.

  5. Sincronización con el repositorio remoto

    6. Con git fetch se descargan los cambios del repositorio remoto antes de aplicarlos al trabajo actual, para ver que cambios se han hecho. git pull obtiene y aplica los cambios remotos a nuestra rama actual. Para realizar la acción inversa (mandar al repositorio remoto) se usa git push, para enviar nuestros commits locales al repositorio remoto.

  6. Integración de cambios

    7. Este proceso se basa en combinar los cambios de diferentes ramas a la principal. git merge incorpora los cambios de otra rama a la actual. Lo que puede causar conflictos si se modificaron las mismas líneas de código en ambas ramas, asunto que se debe resolver de forma manual.
